# 04. 评测、Tracing、安全、MCP 与部署

## Eval

Agent 系统必须能被评测，否则无法判断 prompt、模型、检索、工具链是否真的变好。

### Golden Dataset

建议每个项目至少维护：

- 用户问题
- 期望答案
- 期望引用
- 期望工具调用
- 禁止行为
- 难度标签
- 失败原因标签

### Retrieval Eval

关注：

- recall@k
- MRR
- 命中文档是否正确
- chunk 是否包含答案
- metadata filter 是否生效
- rerank 后相关性是否提升

### Answer Eval

关注：

- 答案是否正确
- 是否基于证据
- 引用是否准确
- 是否在资料不足时拒答
- 格式是否符合 schema

### Tool Calling Eval

关注：

- 是否选择正确工具
- 参数是否正确
- 是否越权调用
- 是否执行 dry run
- 敏感操作是否触发确认

### Trajectory Eval

关注 Agent 的完整行动轨迹：

- 计划是否合理
- 工具调用顺序是否合理
- 是否重复调用无效工具
- 是否能从工具错误中恢复
- 是否在必要时请求人工确认

## Tracing

每次请求至少记录：

- trace_id / user_id / session_id
- prompt 版本和模型版本
- 输入、输出、结构化结果
- tool call 名称、参数、结果摘要、耗时、错误
- retrieval query、命中文档、分数、引用
- token 用量和成本
- 用户反馈

Tracing 的目标不是堆日志，而是能回答：为什么这个 Agent 给出这个答案，错在哪里，如何复现。

## 安全与权限

### Prompt Injection

风险：用户或外部文档让模型忽略系统规则、泄露数据、调用危险工具。

防护：

- 系统规则和外部内容明确分层。
- 文档内容只作为数据，不作为指令。
- 工具调用必须经过程序权限校验。
- 高风险操作必须人工确认。

### 权限与 RBAC

不要让模型决定权限。模型只能提出工具调用请求，程序必须检查：

- 用户是谁。
- 是否有该工具权限。
- 是否能访问目标资源。
- 是否需要审批。
- 是否超过频率或成本限制。

### Sandbox

需要执行代码、文件、浏览器或系统命令时，应使用沙箱：

- 限制文件路径。
- 限制网络访问。
- 限制执行时间。
- 限制 CPU / 内存。
- 记录所有输入输出。

### PII 与数据脱敏

- 日志中避免记录身份证、手机号、邮箱、密钥。
- 训练 / eval 数据集需要脱敏。
- 工具结果返回给模型前做最小化裁剪。

## MCP 生产化要点

- 工具 schema 要稳定，字段语义要清晰。
- 错误返回要可读，便于模型恢复。
- 工具结果要摘要化，避免上下文爆炸。
- server 侧执行权限校验。
- Inspector 调试通过后再接入 Agent。
- 对 destructive tool 默认禁用，必须确认后执行。

## 部署 Checklist

- Dockerfile 和 Compose 可运行。
- 环境变量集中管理，密钥不入库。
- 数据库迁移可重复执行。
- 健康检查、日志、监控、告警齐全。
- 有 rate limit、并发限制、超时和重试。
- 有 token 成本统计。
- 有 eval 回归流程。
- 有审计日志。
- 有备份和回滚方案。
