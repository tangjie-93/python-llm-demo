# 03. 工具调用与业务系统集成

## 学习目标

这一阶段的目标是让 Agent 从“回答问题”升级为“执行任务”。核心不是让模型拥有更大权限，而是让模型在受控工具集合里做选择。

## 1. 工具是什么

工具是 Agent 可以调用的外部能力，例如：

- 查询数据库
- 搜索文档
- 调用 CRM API
- 创建工单
- 发送邮件草稿
- 读取文件
- 运行浏览器自动化

工具由程序执行，模型只负责决定是否调用以及生成参数。

## 2. 工具 Schema

工具 schema 描述参数结构。它应该尽可能明确。

示例：

```json
{
  "name": "search_customer_orders",
  "description": "Search customer orders by customer id, email, or order status.",
  "parameters": {
    "type": "object",
    "properties": {
      "customer_id": { "type": "string" },
      "status": {
        "type": "string",
        "enum": ["pending", "paid", "shipped", "cancelled"]
      }
    },
    "required": ["customer_id"]
  }
}
```

设计原则：

- 参数少而清晰
- 使用 enum 限制取值
- 避免让模型传自由 SQL
- 复杂操作拆成多个小工具
- 写操作必须支持 preview

## 3. 工具权限

每个工具都需要权限策略：

- 谁可以调用
- 什么参数范围允许
- 是否需要用户确认
- 是否需要管理员审批
- 是否写入审计日志

工具风险分级：

| 风险 | 示例 | 策略 |
| --- | --- | --- |
| 低 | 查询天气、查询公开文档 | 可直接执行 |
| 中 | 查询客户订单、生成邮件草稿 | 鉴权 + 日志 |
| 高 | 发邮件、改订单状态 | preview + 用户确认 |
| 极高 | 删除数据、执行命令 | 默认禁用或强审批 |

## 4. Dry Run

Dry run 是在真正执行前生成预览。

适合：

- 发邮件前展示邮件内容
- 创建工单前展示字段
- 修改数据库前展示 diff
- 提交代码前展示 patch

流程：

1. Agent 生成操作计划
2. 工具返回 preview
3. Vue 页面展示确认弹窗
4. 用户确认
5. 后端执行真实操作
6. 写入 audit log

## 5. 工具错误处理

常见错误：

- 参数缺失
- 参数格式错误
- 权限不足
- API timeout
- 业务系统返回错误
- 结果为空
- 返回内容过长

处理原则：

- 错误结果要结构化
- 不要把完整堆栈直接给模型
- 可恢复错误可以让模型重试
- 权限错误不能让模型绕过
- 工具超时要有明确兜底

## 6. Tool Result Summarization

工具返回可能很长，例如数据库查询 100 条记录。不要直接塞给模型。

建议：

- 工具层先分页
- 后端先摘要
- 只返回与任务相关字段
- 原始结果保存在数据库，前端按需查看

## 7. CRM / OA Agent 项目

功能：

- 查询客户资料
- 查询订单
- 创建跟进任务
- 生成邮件草稿
- 修改客户状态
- 生成销售日报

Vue 页面建议：

- 左侧会话区
- 右侧客户信息面板
- 工具调用 Timeline
- 操作确认 Drawer
- 审计日志 Tab

后端建议：

- FastAPI 定义工具 API
- Pydantic 校验参数
- PostgreSQL 保存业务数据和审计日志
- Redis 保存短期任务状态

## 8. 学习路线

1. 先实现 3 个只读工具
2. 再实现 1 个写入工具的 dry run
3. 加入用户确认
4. 加入权限判断
5. 加入 audit log
6. 加入工具失败重试
7. 在 Vue 页面展示工具调用轨迹
