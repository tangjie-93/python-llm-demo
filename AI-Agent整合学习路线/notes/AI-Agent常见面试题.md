# AI Agent 常见面试题（含回答要点）

> 适用：AI Agent / LLM 应用工程师、算法工程师、后端工程师转 Agent 岗。  
> 用法：先用自己的项目经历回答，再用本页的“回答要点”补齐设计取舍。面试官真正想听到的，往往是**为什么这样设计、失败时怎么办、如何证明它有效**。

## 一、30 秒自我介绍模板

“我做过面向 `<业务场景>` 的 Agent 系统。核心链路是 `<意图识别/规划>` → `<检索或工具调用>` → `<结果校验>` → `<最终答复或人工确认>`。我重点解决了 `<一个真实难题：例如工具误调用、检索不准、长任务中断>`，通过 `<方案>` 将 `<指标>` 从 `<改造前>` 提升到 `<改造后>`。系统上线后还通过 `<日志、追踪、离线集、线上反馈>` 持续评测。”

将尖括号替换成你的真实项目；没有线上数据时，诚实说明是离线评测或压测结果。

---

## 二、基础概念与架构

### 1. 什么是 AI Agent？它和聊天机器人、工作流有什么区别？

**回答要点：** Agent 通常由模型、指令、工具、状态/记忆和控制循环组成。聊天机器人主要生成文本；工作流按预定义路径执行；Agent 会基于当前状态决定下一步动作。生产系统常采用“**确定性工作流包住有限自主 Agent**”，而不是让模型无限自由循环。

### 2. Agent 的核心组成有哪些？

**回答要点：**

- 模型：理解、推理、选择动作、生成答复；
- 指令与策略：角色、边界、输出格式、何时升级人工；
- 工具：检索、数据库、业务 API、代码执行等；
- 状态与记忆：当前任务上下文、会话历史、长期偏好；
- 编排器：路由、重试、停止、审批、异常处理；
- 可观测与评测：trace、指标、样本集和回归测试。

### 3. 什么时候不该使用 Agent？

**回答要点：** 输入、步骤和输出都稳定时，普通函数/固定工作流更便宜、更可预测；高风险动作又无法审计或审批时，也不应完全自动化。先从最简单的可行方案开始，只有当规则分支爆炸、任务路径确实不确定时再引入 Agent。

### 4. ReAct 是什么？优缺点是什么？

**回答要点：** ReAct 将“推理/决策—行动—观察”交替进行：模型根据状态选择工具，读取结果后继续或作答。优点是适合多步、信息不完整的任务；缺点是轮次多、延迟和成本高，且可能陷入循环或被错误观察误导。工程上要设置最大步数、总时限、预算和明确终止条件。

### 5. Plan-and-Execute 和 ReAct 如何选择？

**回答要点：** ReAct 边观察边决策，适合探索性任务；Plan-and-Execute 先生成计划再执行，适合可分解的长任务，便于展示进度和审批。实际常组合：先高层规划，每步执行后根据证据重规划；不要把初始计划当作不可改变的真理。

### 6. 单 Agent 和多 Agent 如何取舍？

**回答要点：** 单 Agent 上下文短、调试简单，优先选它。只有角色专业化、工具权限隔离、并行子任务或独立审查确有收益时再拆分。多 Agent 的代价是上下文传递、协议设计、循环对话、成本和故障定位都会更复杂。

### 7. 常见的多 Agent 协作模式有哪些？

**回答要点：** Router → Specialist（按意图分发）、顺序流水线、主管/执行者、并行研究后汇总、生成者—审查者。要明确每个角色的输入/输出 schema、权限、状态所有权和谁有最终决策权。

### 8. 如何设计一个客服 Agent 的总体架构？

**回答要点：** 入口做身份与限流；路由节点分到 FAQ/RAG、订单查询、工单等；读操作可自动执行，退款、取消订单等写操作必须二次确认；工具结果经过校验和脱敏；不可恢复异常进入人工队列；全链路记录 trace、工具参数摘要、耗时和结果。

## 三、Tool Calling 与系统集成

### 9. Function/Tool Calling 的完整链路是什么？

**回答要点：** 向模型提供工具名称、描述和 JSON Schema；模型返回结构化调用意图；服务端校验参数、鉴权并实际执行；将可信的工具结果回传给模型；模型再生成最终回答或下一步调用。关键点：**模型提出请求，服务端才拥有执行权**。

### 10. 如何写好工具定义（schema 和 description）？

**回答要点：** 工具名用明确动词；描述清楚适用/不适用条件；参数少且类型、枚举、必填项明确；对金额、日期、ID 设置格式约束；返回稳定、可机器消费的结构。避免“万能 SQL”“任意 HTTP 请求”这类权限过大的工具。

### 11. 模型生成的工具参数不合法，怎么办？

**回答要点：** 不直接执行。先做 JSON Schema、业务规则和权限校验；可修复错误时给模型返回简短的字段错误让它重试；超过次数则向用户澄清或降级人工。要把校验放在服务端，不能只依赖 prompt。

### 12. 工具调用失败如何设计重试？

**回答要点：** 区分可重试（超时、限流、临时 5xx）和不可重试（鉴权失败、参数错误、业务拒绝）。对前者用指数退避、抖动和最大次数；写操作必须设计幂等键，避免“重试导致重复扣款/重复建单”。

### 13. 如何让 Agent 调用有副作用的工具更安全？

**回答要点：** 最小权限、读写分离、风险分级；写操作展示影响范围并要求用户确认；关键操作引入人工审批；服务端做租户隔离、审计日志、幂等控制和回滚/补偿方案。

### 14. Agent 如何访问数据库？

**回答要点：** 不让模型直连生产库。提供受限查询工具或语义层，使用只读账号、参数化查询、表/字段白名单、行级权限和结果上限；对于分析场景可先生成 SQL，再由规则/解析器校验后在隔离环境执行。

### 15. 如何处理第三方 API 的限流和不稳定？

**回答要点：** 超时、熔断、重试、缓存、队列和降级缺一不可；把错误分类返回给 Agent，而不是把原始堆栈塞回上下文。对用户说明“暂时无法查询”，并给出重试或人工入口。

## 四、RAG、记忆与上下文

### 16. RAG 和 Agent 的关系是什么？

**回答要点：** RAG 是为模型补充外部知识的能力，Agent 是选择何时检索、检索什么、是否调用其他工具的控制系统。简单问答可只用 RAG；需要在检索、查询系统、写入工单等动作间决策时，RAG 可以成为 Agent 的一个工具。

### 17. 如何提高 RAG 的召回和答案质量？

**回答要点：** 从数据清洗、合理切块、元数据过滤、混合检索（关键词+向量）、重排、查询改写、上下文压缩和引用约束逐层优化。评估时拆开看“是否检到正确证据”和“是否根据证据作答”，不要只盯最终回答。

### 18. 什么是短期记忆和长期记忆？

**回答要点：** 短期记忆是当前任务/会话状态，如已执行步骤和用户刚提供的信息；长期记忆是跨会话的稳定事实或偏好。长期记忆要有写入门槛、来源、置信度、过期策略和用户删除能力，避免把模型猜测保存成事实。

### 19. 长对话如何控制上下文窗口和成本？

**回答要点：** 保留系统指令、最近若干轮和结构化任务状态；对较早对话做摘要；按需从记忆库检索；工具返回做字段裁剪和摘要。必须评测摘要是否丢失关键约束，而不是只看 token 变少。

### 20. 如何避免 Agent 被过期或错误知识误导？

**回答要点：** 给文档加来源、版本、生效时间和权限元数据；检索时过滤；要求引用证据；对高风险结论二次验证或转人工；建立知识更新和失效流程。

## 五、状态机、可靠性与可观测性

### 21. 为什么需要状态机/图编排，而不只是 while 循环？

**回答要点：** 图编排显式表达节点、条件分支、循环、错误边和终止点，能持久化状态、从 checkpoint 恢复，并使测试和调试更可控。循环并非坏事，但要有可观测的状态迁移和严格边界。

### 22. 什么是 checkpoint？如何用于长任务？

**回答要点：** checkpoint 保存可恢复的任务状态，如任务 ID、已完成步骤、工具结果引用、待审批动作和版本信息。每个可重试边界后落盘；恢复时先验证状态版本和幂等性，避免重复执行已产生副作用的步骤。

### 23. 如何防止 Agent 无限循环？

**回答要点：** 限制最大步数、总时长、最大 token/费用；检测重复工具调用和状态无进展；定义成功、失败、需澄清和转人工的终止条件；达到限制时输出可解释的降级结果。

### 24. 如何做 Agent 的 tracing？

**回答要点：** 为一次用户请求分配 trace ID，记录模型调用、prompt/版本、token、延迟、工具名、参数摘要、结果状态、状态迁移和最终结果；敏感内容脱敏。trace 要能回答：失败在哪一步、为何选择该工具、成本花在哪里。

### 25. Agent 线上应关注哪些指标？

**回答要点：** 任务成功率、人工转接率、工具成功率、重试率、平均轮次、端到端延迟、token/请求成本、用户满意度与安全拦截率。按意图、工具、模型版本和用户分层，才能定位退化来源。

### 26. 如何处理模型输出格式不稳定？

**回答要点：** 优先使用结构化输出/函数调用；服务端解析、schema 校验与纠错重试；对外部协议建立适配层；关键结果宁可失败降级，也不要“猜一个字段”继续执行。

## 六、评测与优化

### 27. 如何评测一个 Agent 是否“好”？

**回答要点：** 先定义任务目标和失败成本，再构建覆盖真实意图、边界条件、对抗样本和历史失败案例的评测集。按层评估：路由正确、工具选择、参数正确、证据质量、最终答复和安全合规；同时测延迟与成本。

### 28. 离线评测集怎么构建？

**回答要点：** 从脱敏真实日志、专家编写场景、历史事故和合成边界样本获得；标注期望工具、关键事实、不可做动作和可接受答案。划分开发集/保留集，版本化管理，避免用同一批题反复调参后“虚高”。

### 29. LLM-as-a-Judge 是否可信？

**回答要点：** 它适合规模化、模糊质量的辅助评估，但不是绝对真相。需要清晰 rubric、少量人工标注校准、与规则指标/人工抽检交叉验证，并监控评审模型变更带来的漂移。

### 30. 如何做 prompt 或模型版本的 A/B 测试？

**回答要点：** 固定或分层抽样流量，预先定义主指标和安全护栏；比较成功率、成本、延迟、转人工率及关键失败类型；设置停止条件。先在离线回归集通过，再小流量灰度，不以“主观感觉更好”上线。

### 31. Agent 回答错了，排查顺序是什么？

**回答要点：** 用 trace 分层定位：用户意图理解错？路由错？检索没找到？工具选择/参数错？工具返回错？模型没有遵循证据？再针对根因改数据、工具、状态/规则或提示词。不要一看到错误就只改 prompt。

### 32. 怎样优化成本和延迟？

**回答要点：** 减少不必要轮次和上下文，缓存稳定结果，工具并行化，简单子任务路由到小模型，复杂/高风险任务升级强模型；将大工具结果压缩。优化后须验证成功率和安全性没有下降。

## 七、安全、治理与人工介入

### 33. 什么是 prompt injection？如何防御？

**回答要点：** 不可信文本（用户输入、网页、文档、工具结果）试图改变系统指令或诱导泄露/执行危险动作。防御的核心不是“再写一句忽略攻击”：区分可信与不可信数据、最小工具权限、服务端策略校验、敏感操作审批、输出过滤、隔离执行和审计。

### 34. 如何避免数据泄露？

**回答要点：** 身份认证、租户和行级授权、按工具再鉴权、最小化发送给模型的数据、PII 脱敏、加密、保留期限和访问审计。尤其要防止“模型知道一个用户，就能调用工具读另一个用户数据”。

### 35. Human-in-the-loop 应放在哪些位置？

**回答要点：** 高金额/高影响写操作、低置信或证据冲突、政策/法律/医疗等高风险建议、越权请求、模型反复失败，以及抽样质检。审批界面应呈现建议动作、证据、影响范围和可选替代方案。

### 36. 如何设计 Agent 权限模型？

**回答要点：** 把“谁在请求”“Agent 当前角色”“操作对象”“动作类型”“风险等级”都纳入授权；采用最小权限、短期凭证、读写分离和工具白名单。模型不是安全边界，授权必须由可信服务端实施。

### 37. 如何应对幻觉？

**回答要点：** 幻觉不能仅靠提示词消除。对事实性任务，要求检索/工具证据和引用；对无法确认的信息明确说不知道；对关键字段用规则或外部系统校验；把“拒绝编造”纳入评测集。

## 八、项目深挖与系统设计题

### 38. 讲一个你做过的 Agent 项目，最难的问题是什么？

**回答结构（STAR）：** 场景与目标 → 原方案和失败现象 → 通过 trace 找到的根因 → 改造方案与取舍 → 量化结果 → 尚未解决的限制。准备一个可靠性案例和一个效果优化案例，比泛泛讲技术栈更有说服力。

### 39. 设计一个“自动报销审核 Agent”。

**回答要点：** OCR/结构化提取 → 规则预检（票据、金额、重复报销）→ 检索公司制度 → 风险分类 → 低风险自动通过、中风险补材料、高风险人工审核 → 写入时幂等与审计。强调财务动作不能由模型直接决定，模型适合解释、归类、补全信息。

### 40. 设计一个“网页调研并生成报告”的 Agent。

**回答要点：** 先限定主题、可信来源和时间范围；并行检索不同子问题；提取结构化主张及 URL；做来源去重、冲突检测和引用覆盖检查；最后生成带引用的报告。设置单任务预算、抓取限制和无法验证时的明确标记。

### 41. 如何将一个 demo Agent 产品化？

**回答要点：** 从“能跑”补齐为：身份与权限、可靠工具层、状态持久化、异步队列、观测/告警、离线回归、灰度发布、成本治理、隐私合规和人工兜底。面试时请结合你亲自完成的一个改造细讲。

### 42. 如果让你从零开始，前两周会做什么？

**回答要点：** 第一步不是选框架，而是确认用户任务、成功标准、风险边界和可用数据/工具；随后做最小单 Agent/工作流原型和 20–50 条评测集；建立 trace；在离线结果达标后接入真实工具并小流量验证。框架只是实现细节。

## 九、面试前速查清单

- 准备 1 个端到端项目故事：架构图、一次故障、一次指标提升；
- 能画出“用户 → 编排器 → 模型/工具/RAG → 校验 → 审批/答复”的链路；
- 对每个工具说清：输入、权限、超时、重试、幂等、失败降级；
- 准备一套评测指标与 3 个失败案例；
- 不夸大模型能力：明确哪些由模型决策，哪些必须由规则和服务端保障。
